18) Detailed User Flows (End-to-End)
A) Viewer opens app → finds a live → chats → gifts

Open app → Home feed loads live rooms

Tap a room → join room

Stream plays + chat loads

Viewer sends chat message / reacts

Viewer sends gift:

Select gift → confirm → coins deducted → gift event shows for everyone

Viewer follows host → follow count updates

Viewer leaves room

Acceptance criteria

Home shows at least: room thumbnail, host name, viewers count, LIVE status

Entering a room loads stream in < 2–4 seconds on decent network

Chat message appears in room within 300–800ms typical

Gift deducts coins exactly once (no double-charging on retries)

B) Host goes live → moderates chat → ends stream

Host taps “Go Live”

Set title + category + language + (optional) tags

Start stream → room is visible in discovery instantly

Host can:

pin message

mute/kick/ban user

assign moderator

End live → stream stops → room removed from feed

Acceptance criteria

Host cannot start live if account is suspended

Host sees viewer count and recent gifts

Ending live saves a session summary record (duration, peak viewers, gifts total)

C) Private call request → paid 1:1 video call (per-minute)

Viewer taps “Private Call”

App displays rate: X coins/min

Viewer confirms request

Host receives request modal: Accept / Decline

If accepted:

call connects

billing starts when connection established

Every minute:

coins deducted

host earnings ledger updated

Warning triggers if viewer balance < 1 minute remaining

Auto-end when insufficient funds

Edge cases (must handle)

Viewer disconnects temporarily → grace period (e.g., 15–30s) then end

Host declines → no charge

Call never connects → no charge

Deduction retry protection (idempotency key per billing tick)

Acceptance criteria

Billing begins only after both users are connected

Viewer sees ticking timer + current spent amount

Auto-end occurs when balance < required rate for next tick

D) Private call request → paid 1:1 video call (per-session)

Viewer selects “Per Session”

App shows fixed price: Y coins/session

Viewer confirms

Host accepts

Deduct Y coins immediately when call connects (or pre-authorize/hold then capture on connect)

Call runs until user ends OR session max time reached (if enabled)

Edge cases

If host accepts but never connects → release hold (refund)

If viewer ends immediately after connect → still charged (policy decision, default yes)

Acceptance criteria

Only one charge per session call_id

Transaction ledger shows reference to call_id

21) Billing Logic (Critical Spec)
Per-minute billing algorithm (recommended)

Tick interval: 60 seconds (or 10 sec prorated for fairness)

On call:connected:

store billing_started_at

Every tick:

charge viewer

credit host

record ledger transactions

Stop ticking when:

call ended

viewer insufficient funds

disconnect > grace period

Idempotency

Each tick has tick_id = call_id + minute_index

If server retries, it must not double-charge if tick_id already exists in ledger

Refund policy (MVP simple)

If call never connected: full refund / no charge

If connected: no refunds except admin overrides
